# Location_Service iOS native side (task 8.1)

Implements the JS-side spec in `packages/native/src/location/NativeLocationService.ts`.

## Wiring into the iOS xcodeproj

We use Expo bare React Native, so `ios/` and `android/` directories are
regenerated by `expo prebuild` from `app.config.ts` plus the local
config plugins. The Location_Service sources do not move â€” they live in
this directory permanently and are pulled into the generated xcodeproj
through autolinking.

Two options for autolinking:

1. **CocoaPods spec** (preferred). The `Tramio.podspec` snippet at
   `packages/native/ios/Tramio.podspec` declares the Tramio native
   modules as a single in-tree pod. Expo's autolinker picks pods up from
   `node_modules/@tramio/native/ios/` automatically when the package is
   resolved through the workspace. Until task 1.3 wires the workspace
   resolution, a one-line `Podfile` hook in the generated
   `ios/Podfile.properties.json` points at this directory; the snippet
   below documents the contract:

   ```ruby
   pod 'Tramio', :path => '../packages/native/ios'
   ```

2. **Direct file references** (fallback). If autolinking isn't wired up
   yet, the native source files in
   (`Location/`, `Audio/`, `Tts/`) can be added to the generated
   xcodeproj as a Compile Sources build phase in the main app target.
   This is brittle across `expo prebuild --clean` runs, so option 1 is
   the long-term plan.

## Native pipeline responsibilities

| Stage | Owner | Rule | Source |
|-------|-------|------|--------|
| 1. Accuracy gate | Native (this module) | reject if `accuracyM > 50` | Req 5.1 |
| 2. Spike rejection | Native (this module) | reject if `haversine(prev, curr) / dt > 33.33 m/s` | Req 5.2 |
| 3. Smoothing | JS | EMA over last 3 accepted updates | Req 5.5 |
| 4. Dwell | JS | candidate POI must have `smoothed âˆˆ geofence` for `dwellSec` | Req 5.3 |
| 5. Direction filter | JS | heading projected onto route tangent must be within tolerance | Req 5.4 |

The native side is therefore deliberately thin: it owns accuracy + spike
rejection and the iOS-specific 18-region sliding window because both are
about battery / OS limits. Everything that needs to be unit-testable
without standing up CoreLocation lives in JS.

## Mode â†’ CLLocationManager mapping

| Mode | Configuration |
|------|---------------|
| `idle` | stop continuous updates; no SLC; no region monitoring |
| `standby` | `startMonitoringSignificantLocationChanges` |
| `tour-bg` | SLC + region monitoring (sliding window of 18 nearest POIs) |
| `tour-approach` | `desiredAccuracy = kCLLocationAccuracyBest` + `startUpdatingLocation` for the approach window; SLC + regions stay armed |
| `reconcile` | same as `tour-approach`, used after Dead_Reckoning to re-acquire two clean fixes |

`onAccuracyChanged` fires when the module enters / leaves
`tour-approach` or `reconcile` so the UI can drive the user-visible
high-accuracy indicator (Req 11.5).

## Sliding region window (the 20-region cap)

iOS limits an app to monitoring at most 20 simultaneous regions. Long
routes can have many more POIs than that, so the JS layer hands us the
full geofence list via `armGeofences(...)` and we maintain an
18-of-N sliding window keyed on great-circle distance from the most
recent accepted fix. Two slots are intentionally left free for transient
reconciliation regions per design.md "iOS region monitoring limited to
20 active regions".

## Notes for task 8.7 (instrumented smoke tests)

- Cover the accuracy gate with a fake `CLLocation` whose
  `horizontalAccuracy = 75`. Expect a single `onRejected` with
  `reason="accuracy"`.
- Cover spike rejection by feeding two `CLLocation`s 100 m apart with
  a 1 s `timestamp` delta. Expect `onRejected` with `reason="spike"`.
- Cover the sliding window by arming 25 geofences and asserting the OS
  region count never exceeds 20 (`manager.monitoredRegions.count`).
- Cover the high-accuracy indicator by transitioning
  `setMode("tour-bg") -> setMode("tour-approach") -> setMode("tour-bg")`
  and asserting `onAccuracyChanged` fires twice with the expected
  `highAccuracy` values.
