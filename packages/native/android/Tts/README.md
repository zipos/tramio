# TramioTtsEngine (Android)

Android native side of the `TTS_Engine` turbo module declared in
`packages/native/src/tts/NativeTtsEngine.ts`. Wraps
`android.speech.tts.TextToSpeech` (Req 15.1) and exposes the JS-side
spec's `speak` / `pause` / `resume` / `stop` methods plus the unified
playback event stream.

## Files

- `TramioTtsEngineModule.kt` — `ReactContextBaseJavaModule` subclass.
  Owns the lazily-initialized `TextToSpeech` instance, the
  `UtteranceProgressListener`, the voice-resolution chain, and the
  audio-focus listener piggybacked on the system `AudioManager`.
- `TramioTtsEnginePackage.kt` — `ReactPackage` registering the module
  under the bridge name `TramioTtsEngine`.

## Voice resolution

Resolves `(language, region)` against `TextToSpeech.getVoices()` and
the platform's per-language defaults following the documented fallback
chain in `packages/native/src/tts/resolveVoice.ts`:

1. Find a `Voice` whose `Voice.getLocale()` reports the requested
   `(language, region)` (BCP-47 components).
2. Find a `Voice` whose locale's language matches the requested
   language (any region).
3. Use the platform default for the requested language by calling
   `TextToSpeech.setLanguage(Locale)` and accepting whatever voice
   the engine selects.
4. Use the platform default for the bundle's `defaultLanguage` by
   calling `TextToSpeech.setLanguage(Locale(defaultLanguage))`.

Each fallback step that misses logs a non-fatal warning via
`Log.w(TAG, ...)` so the failure is visible in `logcat` without
crashing playback (Req 9.4). If even step 4 misses, the synthesizer
is left to fall through to its built-in any-voice path.

## Event shapes

Mirrors `Audio_Service` (design.md) and the iOS `TramioTtsEngine` so
the engine's command translator can wire either backend without
branching:

- `onPlaybackFinished` — `{ segmentId: string }`. Emitted from the
  `UtteranceProgressListener.onDone`, `onError`, and `onStop`
  callbacks. The task brief states `onError` and `onDone` are not
  distinguished — the engine treats either as a finish event
  (Req 1.7 release-on-end).
- `onFocusLoss` — `{}`. On Android these events are routed through
  `TramioAudioServiceModule`'s focus event stream at the engine level
  (task 13.1 wiring); the native side here does not emit them
  directly because `AudioManager` does not expose a passive
  focus-observation API. The listener API is preserved for parity
  with the iOS module.
- `onFocusRegain` — `{}`. Same routing as `onFocusLoss`.

## Pause semantics

`TextToSpeech` does not expose a true pause primitive. `pause` calls
`engine.stop()` and the JS-side wrapper relies on the engine to
re-issue `speak` on focus regain. This is documented in
`NativeTtsEngine.ts` ("re-synthesizing from the Markdown source from
scratch on resume is acceptable for the MVP").

## Audio focus

Per the task brief ("do not own a separate focus request — observe via
the same BroadcastReceiver path as Audio_Service or share the focus
manager"), this module does NOT register an `AudioFocusRequest`.
`TramioAudioServiceModule` (task 8.4) is the single owner of focus for
the process. The JS-side `onFocusLoss` / `onFocusRegain` listeners
exposed by `NativeTtsEngine` exist for shape-parity with the iOS
module; on Android they are routed through Audio_Service's event
stream at the engine level (task 13.1 wiring).

The Android `AudioManager` does not expose a passive
focus-observation API: an `OnAudioFocusChangeListener` only receives
callbacks when its associated focus request is the focus owner, which
would race `TramioAudioServiceModule`. If a future Android version
exposes a real "register focus listener without requesting focus"
primitive, the focus events can be emitted directly from the native
side here without changing the JS-side spec.

## Wiring

Once the consuming Expo bare project's `android/` directory exists
(generated by `expo prebuild`), `TramioTtsEnginePackage` is added to
`MainApplication.kt`'s `getPackages()` list, exactly the same way
the upcoming Audio / Location packages will be wired in task 13.1.

The minimum SDK target is 21 (Android 5.0 Lollipop), matching the
React Native baseline. `AudioFocusRequest` (API 26+) is gated behind
`Build.VERSION.SDK_INT` so the legacy
`AudioManager.requestAudioFocus(listener, ...)` path is used on
older devices.
